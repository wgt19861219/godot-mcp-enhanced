# godot-mcp-enhanced 问题与改进分析报告

> 分析日期：2026-04-29
> 测试项目：CardGame（Godot 4.5.1/4.6, GL Compatibility, 960x640, 206 GDScript, 98 场景, 6 Autoload 单例）
> 分析依据：近一周密集使用中暴露的实际问题

---

## P0：架构级问题（影响可维护性）

### P0-1: GodotServer.ts 巨石文件（2032 行）

**现状**：34 个工具的注册（188-652 行）和分发（676-1952 行）全在一个 switch-case 里。

**问题**：
- 任何改动都要在 2000 行文件里定位
- merge conflict 风险高
- 无法独立测试单个工具

**建议**：拆分为独立工具模块：
- `tools/scene-tools.ts` — read_scene、create_scene、add_node、save_scene 等
- `tools/script-tools.ts` — read_script、write_script、edit_script
- `tools/project-tools.ts` — get_project_info、validate_project、list_files
- `tools/runtime-tools.ts` — execute_gdscript、run_project、query_scene_tree
- `godot-server.ts` — 只保留注册+分发框架

### P0-2: godot_operations.gd 巨石文件（1263 行）

**现状**：所有操作函数在一个文件，debug_mode 检查遍布全文（约 30% 代码行是 if debug_mode 块）。

**建议**：
- 抽取 LogHelper 自动处理 debug 前缀
- 将操作拆分为独立 GDScript 文件，主文件只做路由

### P0-3: gdscript-executor.ts autoload 模式类型安全问题

**现状**（第 221 行）：
```typescript
(tempFile as any) = JSON.stringify([tempFile, loaderScenePath, loaderScriptPath])
```

**问题**：把 string 变量重新赋值为 JSON 数组字符串，后面 cleanup 时又解析回来。用 any 绕过类型系统。

**建议**：改用结构体传递多文件信息。

## P1：功能缺陷（影响使用效果）

### P1-4: validate_project 误报

**现状**：在 CardGame 上发现 15 个问题，全部来自 tools/migration/output/ 临时目录。

**建议**：
- 增加 .gdignore / .gitignore 感知
- 增加 exclude_paths 参数
- 默认跳过 tools/、addons/

### P1-5: tscn-parser 数组/字典解析不完整

**现状**（第 76-78 行）：
```typescript
if (trimmed.startsWith('[') || trimmed.startsWith('{')) {
  return trimmed;  // 返回原始字符串，不解析
}
```

**建议**：实现基本的递归解析器，支持 Array、Dictionary、Vector2 等常见类型。

### P1-6: Windows 截图限制

**现状**：Windows 总是使用窗口模式，启动慢且可能干扰工作流。

**建议**：先尝试 headless+opengl3，失败后降级窗口模式。

### P1-7: execute_gdscript 的 _mcp_output 机制

**现状**：用户需手动调用 _mcp_output(key, value)，忘记调用则无结构化输出。

**建议**：snippet 模式自动收集最后一个表达式值，或提供 _result 魔法变量。

## P2：使用体验改进

### P2-8: 无进度反馈

耗时操作无中间反馈，AI 可能超时或误判挂起。

**建议**：超过 5 秒的操作增加心跳日志。

### P2-9: edit_script 行号模式脆弱

行号偏移会破坏文件。

**建议**：增加 anchor 模式（前后文内容定位替换区域）。

### P2-10: 场景编辑缺验证闭环

add_node、save_scene 后无自动验证。

**建议**：save 后自动 validate。
